### 关键设计决策（2026-03-20）

为解决“熟悉扫雷的玩家可不依赖卡牌直接通关”的核心问题，`Phase 3` 采用以下主线改造：

- 将传统地雷从“失败判定点”改造为“怪物显形点”
- 玩家揭示到地雷后进入战斗处理，而非立即失败
- 卡牌从“可选辅助”升级为“核心解题手段”
- 保留扫雷信息层（邻格数字与推理价值），并叠加战术层（卡牌战斗）

## 下一阶段计划（Phase 3，已启动）

`Phase 3` 目标是把“单局可玩”升级为“卡牌主导的可持续 Run 体验”：

- 一局扫雷作为“一幕（Act）”的空间化关卡结构（Act-as-Grid）
- 怪物化扫雷基础规则（Mine -> Monster）
- 回合与资源循环（抽牌 / 能量 / 敌方意图 / 回合结算）
- 战斗结果与失败条件重构（揭雷不即死，失误通过资源与血量结算）
- 关卡推进与难度曲线（Level）
- 事件系统与奖励三选一（Event / Reward）
- Run 状态与基础存档（LocalStorage）
- 玩家成长系统（Perk / Meta，轻量起步）

### Phase 3 子阶段拆分（执行顺序）

1. **P3-A：玩法闭环重构（最高优先级，已完成）**
   - 完成 Mine -> Monster 的最小规则闭环
   - 打通“揭示 -> 战斗处理 -> 结算 -> 继续探索”流程
   - 验证“会扫雷但不会出牌”无法稳定通关
   - 完成状态：
     - 揭雷进入怪物遭遇（带 HP/反击），不再直接判负
     - 卡牌可处理遭遇（首轮战斗卡池），并接入结算
     - 怪物点击交互改为“信息查看”，失败条件改为生命耗尽
     - 遭遇结算后触发回合循环（能量重置 + 手牌补充）
     - 安全格清空后若仍有未处理怪物，禁止直接胜利

2. **P3-B：Run 骨架搭建（进行中）**
   - 本轮已完成（战斗开场）：
     - 首轮战斗卡池接入（输出/防御/控制/资源）
     - 怪物意图、标签克制与状态效果（易伤/烟幕覆盖）接入
     - 起始牌组调整为可稳定击杀结构，支持更明确的战斗决策
     - 能量与抽牌循环首版落地（翻格供能、遭遇保底、每回合恢复、抽牌动作）
     - 起始牌组改为配置驱动（`CONFIG.player.startingDeck`）
     - 牌库/弃牌堆查看界面完成全屏化，并统一为手牌同风格卡牌展示
     - 恢复遭遇阻挡规则：存在当前显形怪物时，常规翻格受阻
     - 烟幕卡改为“2 回合解除目标阻挡”，提供短时探索窗口
     - 表现层第一阶段（Juice Phase 1）：
       - 接入 `howler.js` + `AudioManager`（统一音效入口，缺失音频静默降级）
       - 接入 `EffectsManager`（Overlay Canvas 特效层）
       - 完成高优先级动效：点击脉冲、怪物显形、受击闪白+飘字、阻挡高亮、玩家受伤闪烁
       - 接入关键事件 SFX：翻格/出牌/显形/受伤/击杀/阻挡/胜利/失败
      - `AI-EVT-CORE-01` 已完成：
        - `Game` 主流程接入事件状态机（`hidden -> revealed -> pending -> resolved`）
        - Boss gate 三条件接入运行时判定（发现 Boss / 战斗事件阈值 / 困难或精英次数）
        - gate 阻挡与放行均可复现，且不破坏现有卡牌与战斗交互闭环
      - `AI-EVT-DATA-01` 已完成：
        - 事件数据层定义落地（`eventDefinitions`：类型/子类型/查询）
        - 商店数据层定义落地（`shopDefinitions`：多档位/刷新费用）
        - `CONFIG.events` 参数块落地（事件权重/Boss 门槛/商店配置）
        - 敌人金币收益梯度配置落地（普通 < 困难 < 精英）
      - `AI-EVT-UI-01` 已完成：
        - 事件展示 UI 可用（翻开后可见，含最近事件与时间线）
        - 商店三档位 UI 可用（基础/进阶/精英，支持同幕回访）
        - 商店刷新与金币消耗可用（含不足反馈）
        - 商店购买链路可用（金币消费 -> 卡牌入库）
        - 战斗结算金币奖励已接入 UI 经济闭环
      - `AI-EVT-SHOP-SEM-01` 已完成：
        - 商店从“常驻可进”修正为“商人事件触发后可进”
        - 触发后同一幕内可持续回访，未触发时入口/刷新/商品区统一锁定
        - 未改动 Boss gate 与战斗金币结算主流程
   - 待完成：
  - 关卡推进、事件、奖励三选一最小可用（在 core 层消费新数据定义）
  - Run 过程状态可保存/加载

### 设计方向补充（2026-04-01）

为消除 P3-B 的风险与歧义，本阶段新增以下统一口径：

- 一局扫雷即一幕（Act），`雷` 统一语义为“事件触发点”
- 事件分为三类：敌人事件、NPC 事件、奖励事件
- 事件翻开前类型不可知（可选弱提示），翻开后进入对应交互
- Boss 解锁条件首版固定为：
  - 发现 Boss 节点
  - 完成战斗事件 `>= N`（建议 `N=4`）
  - 至少完成 `1` 次困难或精英战斗
- 商店事件首版能力：
  - 多档位商品
  - 刷新按钮（金币消耗）
  - 同一幕可多次访问
- Boss 动态强化（随幕进度浮动）延后到后续平衡阶段
