# 开发执行规范（Development Guide）

本文件用于开发人员日常执行，包含本地运行、提交流程、检查清单与协作约定。

---

## 1) 本地运行

在项目根目录执行：

```bash
python -m http.server 8080
```

访问：`http://localhost:8080`

说明：

- 本地修改应即时生效；如未生效，先强刷浏览器缓存（`Ctrl + F5`）
- 页面右上角会显示当前版本短哈希（本地优先读取 `.git`，线上读取 `version.json`）

---

## 2) 开发流程（必须执行）

1. 完成功能开发
2. 本地验证通过
3. 执行 Git 提交（commit）
4. 提交后立即推送（push）到远端
5. 默认执行：分支开发完成后，校验通过即合并到 `main` 并 push `main` 触发发布

约束：

- 不允许“改完不提交、不推送”的悬空状态
- 每次工作都应对应可追踪提交，便于部署回溯
- 仅 `main` 分支 push 会自动触发部署；推送到功能分支不会自动发布
- 完成后必须回报 push 证据（远端分支、commit 哈希、关键命令输出）
- 若校验失败，禁止合并到 `main`，必须先回报失败原因和修复计划

---

## 3) 提交前检查清单

- 功能流程可跑通（菜单 -> 开局 -> 用卡 -> 胜负）
- 控制台无明显异常
- 版本号角标显示正常
- 关键交互无回归（目标选择、取消、无目标卡二次确认）
- 移动端基础触控无明显回归

---

## 4) 文档同步要求

当改动涉及功能行为或模块职责时，需同步更新文档：

- 模块变化：更新对应目录 `README.md`
- 阶段变化：更新 `ROADMAP.md`
- 版本记录：更新 `RELEASE_NOTES.md`
- 回归用例：更新 `SMOKE_TEST.md`
- 新人导航变化：必要时更新根目录 `README.md`

文档范围约束（强制）：

- 功能任务默认禁止改动设计文档（除非任务明确要求）：
  - `ACT_STRUCTURE_DESIGN.md`
  - `COMBAT_DESIGN.md`
  - `ENERGY_DRAW_LOOP_DESIGN.md`
  - `CARD_ROLE_MATRIX.md`
  - `EVENT_SYSTEM_DESIGN.md`

---

## 5) 推荐提交信息风格

- `feat:` 新功能
- `fix:` 修复问题
- `docs:` 文档更新
- `refactor:` 结构重构（不改行为）

示例：

- `fix: stabilize card switching in targeting mode`
- `docs: update roadmap for phase 3 transition`

---

## 6) 协作建议

- 尽量小步提交，避免超大变更难以回滚
- 先保证可用性，再做视觉与体验优化
- 关键机制（状态机、卡牌流程、胜负判定）改动必须带回归验证

---

## 7) 多 AI 并行开发规范（新增）

适用场景：将不同模块并行分配给多个 AI 开发。

### 7.1 任务定义规则

- 不再以“工时估算”作为核心字段，改用“依赖关系 + 验收标准”
- 每个任务必须明确“允许修改文件”和“禁止修改文件”
- 每个任务必须包含可直接执行的 Prompt 模板
- 每个任务必须定义 DoD（Definition of Done）

### 7.2 分支与合并规则

- 每个 AI 使用独立分支：`feature/<task-id>`
- 相同文件尽量只分配给一个 AI
- 合并前必须执行一次对应任务的最小验证
- 合并后必须执行 `SMOKE_TEST.md` 核心流程回归

### 7.3 统一回报格式

每个 AI 在完成任务后必须回报：

1. 修改文件列表  
2. 关键实现点  
3. 验证步骤与结果  
4. 未完成项与风险  
5. Git 交付证据（必须）：
   - `git status -sb` 结果
   - `git log -1 --oneline` 结果
   - `git push` 结果（含远端分支）
   - 若需要发布：说明是否已合并并 push 到 `main`

### 7.4 任务板入口

- 多 AI 任务卡与提示词模板：`MULTI_AI_TASK_BOARD.md`
