# 事件系统设计草案（Event System v0.1）

**适用阶段**：`v0.3.x-alpha`（P3-B Run 骨架）  
**目标读者**：策划、程序、测试  
**文档定位**：统一“雷=事件触发点”的规则口径，消除实现歧义。

---

## 1. 设计目标

- 形成完整游玩循环：`翻格子 -> 遭遇事件 -> 事件交互 -> 翻格子`
- 一局扫雷即一幕（Act），Boss 被击败后本幕结束
- 事件翻开前不可知，保留探索与赌博张力
- 通过门槛与经济约束，避免“只拿奖励不打战斗”成为最优

---

## 2. 事件分类（首版）

## 2.1 敌人事件（Enemy Event）

- 普通怪（Easy）
- 困难怪（Hard）
- 精英怪（Elite）
- Boss（独立节点，满足解锁条件后可挑战）

核心交互：

- 使用卡牌处理敌人
- 胜利获得金币收益

## 2.2 NPC 事件（NPC Event）

- 商人事件（Merchant）

核心交互：

- 使用金币购买卡牌
- 提供多档位商品
- 提供刷新按钮（消耗金币）
- 同一幕可多次访问

## 2.3 奖励事件（Reward Event）

- 金币宝箱（获得金币）
- 休息点（恢复生命）
- 卡牌宝箱（三选一拿一）

---

## 3. 事件可见性规则

- 玩家翻开前仅知道该格为“事件格”，未知具体类型
- 翻开后显示具体事件类型并进入对应交互
- 可选弱提示：
  - 仅提示“可能危险/可能收益/可能交易”
  - 不直接展示确定类型

---

## 4. Boss 解锁与幕胜利

## 4.1 Boss 解锁条件（首版拍板）

必须同时满足：

- 发现 Boss 节点
- 完成战斗事件 `>= N`（建议 `N=4`）
- 至少完成 `1` 次困难或精英战斗

## 4.2 幕胜利条件

- 击败 Boss

## 4.3 延后项

- Boss 动态强化（基于战斗完成度）延后到平衡阶段

---

## 5. 经济约束（首版建议）

## 5.1 敌人金币收益

- 普通：`+8`
- 困难：`+14`
- 精英：`+24`

目标：

- 困难/精英收益必须明显高于普通，形成挑战动机

## 5.2 奖励事件建议

- 金币宝箱：`+10 ~ +18`
- 休息点：恢复 `30%` 最大生命
- 卡牌宝箱：普通池权重更高，稀有低权重

## 5.3 商店价格档位

- 低档：`10~16`
- 中档：`18~28`
- 高档：`30~42`
- 刷新费用：`8 -> 12 -> 16`（阶梯并封顶）

---

## 6. 事件状态机（实现口径）

统一状态：

- `hidden`：未翻开
- `revealed`：已翻开，未进入处理
- `pending`：正在交互或待结算
- `resolved`：已完成

推荐字段：

- `id`
- `type`（enemy / npc / reward / boss）
- `subType`（easy / hard / elite / merchant / chest_gold / rest / chest_card / boss）
- `state`
- `position`（row/col）
- `payload`（敌人数据、商店库存、奖励内容）

---

## 7. 验收清单（P3-B）

- 能完成完整循环：翻格 -> 触发事件 -> 处理 -> 继续翻格
- 事件翻开前不可知，翻开后可识别类型
- Boss 解锁门槛生效（含困难/精英约束）
- 商店支持多档位、刷新、重复访问
- 纯绕战路线无法稳定满足 Boss 挑战条件
